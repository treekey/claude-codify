# B. CLAUDE.md 重寫準則

目標：把 CLAUDE.md 收斂成「每個 session 都值得付的 token 成本」的精簡本體，細則外移到 `.claude/rules/` 並以一行索引指向。

> **lite 層級注意**：lite 不建 `.claude/rules/`，下述結構中的「細則索引」段改為兩個內嵌段——「判斷判準」（精選 rubric ≤6 條，必含何時升級與何時停下）與「制度維護」（3 行：行數上限、踩坑就近以註解記錄、複雜度上升時建議 `upgrade standard`）。全檔上限 80 行，其餘規則照用。

## 結構（固定順序）

```markdown
# <專案名>

<兩三句：這個專案是什麼、給誰用、目前階段。>

## 常用指令
<build / test / lint / run 的確切指令，含工作目錄。這是全檔最高價值段落。>

## 硬規則
<絕不可違反的事項，每條一行。例：絕不修改 src/generated/ 下的檔案。>

## 同步耦合
<「改 A 必改 B」清單。例：改 db/schema.sql 必須同時新增 migrations/ 檔案。>

## 細則索引
- 模型調度與派工：.claude/rules/dispatch.md
- 品質與完成判準：.claude/rules/rubrics.md
- 派工模板：.claude/rules/prompt-templates.md
- 制度維護：.claude/rules/maintenance.md
```

## 弱強差異化寫法

同一件事，對強弱模型要用不同粒度書寫。原則：**判斷交給強模型，執行步驟寫給弱模型**。

- 需要判斷的規則 → 寫「觸發條件 + 動作」而非開放式描述。
  - 差：「謹慎處理資料庫變更」
  - 好：「凡 diff 涉及 `db/` 目錄：先跑 `npm run db:check`，失敗即停止並回報，不得嘗試修 migration」
- 每條規則自我測試：**Haiku 只看這一條，能否無歧義地執行？** 不能 → 補判準或範例，或改為「此情況升級處理」（升級路徑見 dispatch.md）。
- 有例外的規則，例外要寫死，不要用「通常」「盡量」：
  - 差：「盡量不要加新依賴」
  - 好：「新增依賴前先問使用者。例外：devDependencies 中的 @types/* 可直接加」

## 刪減判準（收斂重複與過時）

逐條檢視既有內容，符合任一條即刪或外移：

1. **可推導**：模型讀 package.json / 目錄結構就能知道的事（「這是 React 專案」）→ 刪。
2. **過時**：規則指向的檔案、指令、人員已不存在 → 刪，並在 commit message 註明。
3. **重複**：同一規則出現兩次以上 → 留最精確的一條。
4. **敘事**：背景故事、決策過程、道歉性說明 → 外移到 docs/ 或刪除，CLAUDE.md 只留結論。
5. **低頻**：一年用不到一次的深度細節（發版流程細節等）→ 外移到 `.claude/rules/`，索引留一行。
