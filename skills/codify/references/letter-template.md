# G. 給未來 session 的信 — 模板

目標：產出專案的 `.claude/rules/letter.md`。這封信承載「制度檔案裝不下、但未來的模型應該知道」的內容。由 init 模式的立法者（高階模型）撰寫；後續 session 只能改交接段。

## 撰寫判準

- 「三件事」必須是**使用者沒問、其他六份檔案也放不進去、但影響決策**的事。典型類型：
  - 專案的真實優先序（例：「這專案表面上重視測試覆蓋，實際上出貨日期永遠優先，衝突時先問而不是先補測試」）
  - 環境的隱性事實（例：「CI 的 Windows runner 比本機舊一版，本機過不代表 CI 過」）
  - 使用者的協作偏好中未成文的部分（例：「使用者說『看一下』通常是要診斷報告，不是要你修」）
- 每件事寫成「情境 → 該怎麼做」，不寫成感想。
- 找不到三件有價值的事，寧可寫兩件，不要湊數——湊數的條目會稀釋整封信的可信度。

## 模板（`.claude/rules/letter.md`）

```markdown
# 給未來 session 的信

寫於 <日期>，由 <模型名> 在 init 立法時撰寫。
除「交接」段外，本檔僅在 audit 模式或經使用者同意時修改。

## 三件沒人問但你該知道的事

### 1. <一句話>
情境：<什麼時候會遇到>
該怎麼做：<具體動作>

### 2. …
### 3. …

## 制度退化預防

這套制度最可能的死法，按機率排序：
1. <本專案最可能的退化路徑，例：CLAUDE.md 重新膨脹> — 徵兆：<可觀察徵兆>；解法：<動作>
2. …

如果你發現制度已明顯退化：不要就地修補規則（那會加速腐化），
建議使用者跑一次 audit 模式；使用者不在時，把觀察寫入 maintenance.md 踩坑記錄。

## Harness 極限備忘

以下事項拆解與多輪評審也補不了，遇到請直接採取對應動作：
- <專案內已知的模糊題型> → <升級／問使用者／坦言做不到>
- 品味與手感判斷 → 對照 rubric 舉例比較後，交人工裁決

## 交接（後續 session 可改；完成即清除）

<未完項目：現況、下一步、注意事項。無則寫「無」。>
```
