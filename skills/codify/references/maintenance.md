# F. 維護協議模板

目標：產出專案的 `.claude/rules/maintenance.md`，定義制度如何被修改與保養，防止兩種死法：**膨脹**（規則越加越多沒人刪）與**腐化**（規則過時卻繼續被遵守）。

## 1. 檔案修改權限

| 檔案 | 模型可自改？ | 條件 |
|---|---|---|
| `.claude/rules/diagnosis.md` | 可 | 僅追加新觀察，不改歷史記錄 |
| `.claude/rules/letter.md` 交接段 | 可 | session 結束交接時 |
| 踩坑記錄（本檔第 3 節） | 可 | 依格式追加 |
| `CLAUDE.md` 常用指令段 | 可 | 指令實測失效時修正，commit 註明 |
| `CLAUDE.md` 硬規則／同步耦合段 | **不可** | 需使用者同意 |
| `.claude/rules/dispatch.md`、`rubrics.md` | **不可** | 提案寫入踩坑記錄，audit 時裁決 |
| `.claude/rules/prompt-templates.md` | 半 | 可追加新模板，不可改既有模板 |

原則：**執行中的 session 不得修改約束自己的規則**。發現規則有問題，記錄提案，由 audit 模式或使用者裁決。

## 2. 體檢 checklist（audit 模式執行）

建議節奏：每 <N 週|N 個 milestone> 或換用新模型時執行一次。

- [ ] CLAUDE.md 檔尾的層級標記（`<!-- codify: tier=... -->`）是否存在且與實際檔案結構相符？體檢範圍以標記層級為準——低層級本來就沒有的檔案不是退化。
- [ ] CLAUDE.md 是否又超過 150 行？超過的部分外移。
- [ ] 逐條抽查硬規則與同步耦合：指向的檔案／指令還存在嗎？失效即刪。
- [ ] 踩坑記錄有幾條待裁決提案？逐條裁決：採納（改進對應規則）／駁回（註明原因）。
- [ ] 抽最近 3 個 AI 產出的 commit，對照 Rubric 4 檢查：規則有被遵守嗎？沒有 → 是規則不可執行（改規則）還是派工缺三要素（記錄提醒）？
- [ ] `.claude/rules/` 下是否有從未被引用的檔案？確認後刪除。
- [ ] letter.md 的交接段是否有已完成卻未清除的項目？清除。
- [ ] （僅 full 層級）逐一實際觸發每個 hook 確認仍會擋；核對每個 hook 在 CLAUDE.md 硬規則段有對應的 `[hook 強制]` 條文；檢查踩坑記錄中是否有「反覆被同一 hook 擋」的情況（是＝規則或阻擋訊息設計有問題，提案修正而非移除 hook）。

## 3. 踩坑回饋機制

任何 session 踩坑後（規則誤導、規則缺失、規則互相矛盾），依此格式**追加**到本節，不做其他修改：

```markdown
### <日期> — <一句話描述坑>
- 情境：<當時在做什麼>
- 坑：<哪條規則出了什麼問題，或缺了什麼規則>
- 提案：<建議怎麼改，一條以內>
- 狀態：待裁決
```

audit 時逐條裁決，狀態改為「已採納 → <改了哪裡>」或「駁回 → <原因>」。已裁決超過 <N> 條時歸檔到檔尾摺疊區。

## 4. 制度退化的警訊（任一出現，建議使用者跑一次 audit）

- 派工不再附三要素，「大家憑感覺做」。
- CLAUDE.md 被隨手加了三條以上未經裁決的規則。
- 踩坑記錄超過 5 條無人裁決。
- 同類錯誤在踩坑記錄出現第二次（表示上次裁決沒有落實成規則）。
