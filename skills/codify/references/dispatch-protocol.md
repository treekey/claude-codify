# C. 模型調度守則模板

目標：產出專案的 `.claude/rules/dispatch.md`，定義任務如何指派給不同等級的模型、何時升降級、以及執行與驗證的分離。以下是模板與填寫判準；產出時把 <> 內容換成該專案的實際值。

## 1. 派工三要素

任何派工（給 subagent、給較弱模型的 session、給排程任務）必須同時包含三要素，缺一不派：

```markdown
- 目標：<一句話，含完成後世界的樣子，不是動作清單>
- 驗收：<可機械檢查的條件，例：`npm test` 全綠、diff 不超出 src/foo/、輸出含欄位 X>
- 回報格式：<固定結構，例：改動檔案清單 + 每檔一句原因 + 驗收執行結果原文>
```

判準：驗收條件若無法由派出方機械地檢查（跑指令、比對格式），就不算驗收條件，重寫。

## 2. 模型分級與預設指派

| 等級 | 適用 | 例 |
|---|---|---|
| 低（Haiku 級） | 有明確模板可填的機械工作：批次改名、格式轉換、照模板產文件、跑指令回報 | <專案內的實例> |
| 中（Sonnet 級） | 範圍明確的實作與重構、有 rubric 可循的審查、資料整理與初步研究 | <專案內的實例> |
| 高（Opus/Fable 級） | 架構決策、模糊需求定義、跨模組設計、rubric 本身的修訂、品味判斷 | <專案內的實例> |

預設從低往高派：先問「這件事低一級 + 更嚴的驗收條件能不能做」，不能才升級。

## 3. 升降級路徑

**升級觸發（任一成立即升級，不要掙扎）：**
- 同一驗收條件連續失敗 2 次。
- 發現任務需要在兩個以上合理方案間做取捨，且守則與 rubric 沒有給判準。
- 任務範圍在執行中擴大（原估 <N> 檔案，實際涉及超過）。
- 遇到 rubrics.md 標記為「升級處理」的情況。

升級動作：停止修改，把「已嘗試什麼、卡在哪、兩個候選方案」寫成交接段落，再由高一級模型或使用者接手。**禁止帶著不確定繼續硬改。**

**降級觸發：** 高階模型完成設計後，剩餘工作若已可寫成三要素，必須降級派出，不要自己順手做完——高階模型的額度應留給只有它能做的判斷。

## 4. 驗證分離

- 執行者不得自行宣告通過：驗收由另一個 session／subagent 或派出方執行。
- 驗證者只拿到「三要素 + 產出」，不拿執行過程的推理，避免被說服。
- 驗證結果只有三種：通過／不通過（附具體違反項）／驗收條件本身有問題（回報派出方修條件）。

## 5. 重大結論抗辯

觸發情境（任一成立即觸發）：架構決策、bug 根因判定、影響生產環境或資料的變更、安全判斷、無法確認影響範圍的結論。

執行方式：同一訊息平行派出三個獨立反方 subagent，各持一種鏡頭，只拿到結論與證據、不拿推理過程（避免被說服）：

- **反例鏡頭**：構造一個讓結論不成立的具體輸入、情境或程式路徑。拿不出具體反例才算讓結論通過；「感覺怪」不算反駁。
- **代價鏡頭**：假設結論是錯的，列出最壞後果與波及範圍（資料、生產環境、下游模組），然後回答：現有證據的強度撐得起這個代價嗎？撐不起就退回補證據。
- **替代鏡頭**：提出至少一個同樣能解釋現象（或達成目標）的更簡單方案，並逐點比較。原結論說不出勝過替代方案的理由就退回。

裁決判準：
- 三個鏡頭中至少兩個未能推翻，結論才算 confirmed。
- 未通過抗辯的結論只能標註為「未抗辯假設」，不得當作事實陳述或作為後續行動的前提。
- 影響生產環境或資料的重大結論：連續兩輪抗辯無新發現才可收工。

成本判準：抗辯是全制度最貴的機制，僅限上述觸發情境使用，一般任務不得套用。lite 層級不設抗辯，以「明示假設 + 升級詢問使用者」替代。

## 6. Harness 極限標註

以下情況分派與驗證都補不了，明定處理方式：
- 模糊需求（「做得好看一點」）：先由高階模型或使用者收斂成可驗收的描述，才能派工。
- 品味判斷（命名品質、API 手感）：低中階模型只能對照 rubric 舉例比較，最終裁決升級或坦言「此項無客觀判準，建議人工裁決」。
