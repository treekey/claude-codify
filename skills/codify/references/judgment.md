# 立法判斷備忘錄

這份檔案是高階模型（Fable 5，2026-07）在立法情境下判斷方式的快照。
它存在的原因：高階模型不會永遠可用，但它「怎麼想」可以留下來。
**任何模型執行 init 前必讀本檔**；讀完後照著這裡的判準做，遇到本檔未涵蓋的模糊情境，
唯一正確的動作是問使用者或明寫「此處無法確定」，不要用自己的直覺補。

## 一、診斷時怎麼挑「前三項」

- 排序公式：**發生頻率 × 單次代價 × 本次可修性**。每個 session 都會踩的小坑，排在一年一次的大坑前面；今天修不掉的問題再大也只進「其餘觀察」。
- 證據等級由高到低：git log 中的 revert／fix-the-fix 記錄 → 測試指令實際跑不跑得動 → CLAUDE.md 的行數與矛盾 → 目錄結構的可疑處。從高等級證據開始找，找滿三項就停，不要為了湊而降低證據標準。
- 「我看不慣」不是問題。風格偏好（縮排、命名品味）不進診斷；只有可觀察的浪費（token、時間）或錯誤（revert、誤改）才進。

## 二、把判斷改寫成弱模型可執行規則的五個手法

寫規則時逐條套用，套不進去的規則就不該寫成規則：

1. **模糊詞 → 觸發條件＋動作**。
   - 前：「謹慎處理資料庫變更」
   - 後：「diff 涉及 `db/` 時：先跑 `npm run db:check`，失敗即停止回報」
2. **開放判斷 → 封閉選項＋fallback**。弱模型面對「自行判斷」會亂猜；給它 2–3 個選項和一條「都不符合就升級」。
3. **原則 → 反例清單**。「不要過度工程」對弱模型無感；「不要為只有一個呼叫者的函式寫抽象層」有感。原則講不清楚的，就列三個最常見的違反樣態。
4. **程度詞 → 數字**。「小步前進」→「單次 commit 不超過 5 個檔案」。數字寧可略嚴，audit 時可放寬。
5. **每條硬規則配一個「違反的樣子」**。識別比生成容易——弱模型認得出違例，就守得住規則。

## 三、常見專案形態的立法要點

- **Monorepo**：制度立在 root 一套，子專案差異用一張表列（哪個子專案用哪組指令），不要每個子專案各立一套——多套制度必然漂移。
- **沒有測試的 legacy**：不要立「改動必附測試」——做不到的規則會讓整套制度失去公信力，模型學會「規則可以忽略」比沒有規則更糟。改立最低可行驗證：characterization test 的門檻，或「驗證＝這份手動步驟清單」。
- **產生碼（generated code）多的專案**：保護路徑規則是第一優先，其他都可以晚點立。
- **不在 git 裡的專案**：備份規則加倍嚴格（改前必備份、備份名帶日期），並在診斷報告中建議 `git init`——這比任何條文都有效。
- **既有巨大 CLAUDE.md**：先分類（可推導／過時／重複／敘事／低頻）再刪；整理出「建議刪除清單」給使用者過目後才動手，不擅自丟棄可能承載歷史教訓的內容。
- **多語言 repo**：常用指令段按語言分小節，每節首行寫適用目錄。

## 四、層級推薦的邊界判斷

- **訊號衝突時，風險優先於規模**：單人專案但會碰生產環境 → 推薦 standard 並加一個保護 hook（說明這是 standard 的例外配置，記進層級標記後的註解），而不是整套 full。
- **prototype 即將轉正**：立 lite，但在 CLAUDE.md 維護段寫死升級觸發點（「出現第二位協作者或接上 CI 時，跑 /codify upgrade standard」）。
- **使用者反對推薦**：直接接受使用者的選擇，不重複勸說；把你認為的風險用一句話記進產出（維護段或信），僅此而已。

## 五、弱模型立法的已知失敗模式（收尾對抗審查時逐條自檢）

1. **模板照抄**：產出裡殘留 `<...>` 佔位符或範例專案的名字。檢查法：全文搜 `<`、搜 exemplar 裡的虛構專案名。
2. **無證據立法**：規則說不出來源。每條硬規則必須指得出依據——某個檔案、某筆 git 記錄、或使用者的某個回答。指不出來就刪。
3. **過度立法**：把「可能有用」的規則都寫進去。判準：這條規則若不存在，最近一個月內會發生什麼具體壞事？答不出來就不立。
4. **通用最佳實務充數**：「寫好測試」「命名要清楚」放諸四海皆準，不是制度。制度是**這個專案特有**的知識；通用的事模型本來就會。
5. **問題問錯方向**：五個開場問題要問「只有使用者知道的事」——真實優先序、風險偏好、不可動的區域、隱性耦合、歷史教訓。不要問模型自己查得到的事，不要問「你偏好什麼風格」這種可以從程式碼歸納的事。

## 六、對抗審查的攻擊角度（收尾步驟 1 用）

逐條規則問這四個問題，任何一個答不出來就修：

1. Haiku 拿到這條，最可能的**誤解**是什麼？（誤解不出來＝條文夠具體）
2. 這條和哪一條**可能衝突**？衝突時誰贏？（沒寫誰贏＝漏洞）
3. 這條第一次被違反時，**誰會發現**？（沒人會發現＝要嘛加驗證、要嘛降為建議）
4. 這條一年後最可能**因為什麼而過時**？（把答案寫進 audit 的檢查點）

## 七、本檔自身的維護

本檔是快照，不隨專案 audit 更新。只有在使用者取得更強模型、明確要求「重新蒸餾判斷」時才整份重寫；
平時發現本檔的判準有誤，把觀察記進該專案的踩坑記錄，不要直接改本檔。
