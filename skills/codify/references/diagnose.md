# A. 快速診斷

目標：找出這個專案的 AI 協作環境中「最耗 token、最易失焦、最易出錯」的前三項問題，各附一條可立即執行的修法。診斷結果寫入 `.claude/rules/diagnosis.md`。

## 檢查清單（依序執行，每項記錄「通過／問題＋證據」）

### 1. Token 消耗

- [ ] `CLAUDE.md` 是否超過 150 行或 6KB？（超過＝問題，修法：外移細則到 `.claude/rules/`）
- [ ] `CLAUDE.md` 是否有重複或彼此矛盾的規則？（用關鍵字比對，列出行號）
- [ ] 是否有巨大檔案（>2000 行）位於模型常需讀取的路徑上，且無導覽說明？（修法：在 CLAUDE.md 加「這個檔案只讀 X 段」的指引）
- [ ] 是否缺少常用指令速查（build / test / lint 的確切指令）？缺少會迫使每個 session 重新摸索。

### 2. 失焦風險

- [ ] 專案是否有多個入口／子專案，但 CLAUDE.md 未說明「預設在哪個目錄工作」？
- [ ] 是否存在名稱相似、容易改錯的檔案或模組？（列出具體配對，修法：加消歧義規則）
- [ ] 需求常見的模糊點（例如「支援多語系」在此專案的確切含義）是否已有定義？

### 3. 出錯熱點

- [ ] `git log --oneline -30` 中是否有 revert / fix-the-fix 型 commit？指向的區域就是熱點。
- [ ] 是否有「改了 A 必須同步改 B」的隱性耦合（schema 與 migration、enum 與 switch、i18n key 與翻譯檔）？每一組都要寫成明文規則。
- [ ] 測試指令是否能在單機直接跑？若需要特殊環境，是否已寫明替代驗證法？
- [ ] 是否有絕不可動的檔案（產生的程式碼、vendored 依賴、正式環境設定）？

## 輸出格式（`.claude/rules/diagnosis.md`）

```markdown
# 診斷報告 — <日期>

## 問題一：<一句話>
- 證據：<檔案:行號 或指令輸出>
- 影響：<耗 token／失焦／出錯，量化描述>
- 修法：<已執行的具體修正，或指向新規則的位置>

## 問題二：…
## 問題三：…

## 其餘觀察（未達前三但值得記錄）
- …
```

## 判準

- 每個問題必須附「證據」，不得憑印象。找不到證據的疑慮寫進「其餘觀察」。
- 修法必須是本次 session 就能完成的動作；做不到的（例如需要重構 CI）明寫「需使用者決策」。
