# D. 判斷力外化 — rubric 與 checklist 模板

目標：把高階模型的判斷轉成弱模型可執行的判準，寫入專案的 `.claude/rules/rubrics.md`。以下四份是必備 rubric；產出時把 <> 換成專案實際值，並依診斷結果增補專案特有的判準。

## 撰寫規則

- 每條判準必須是「可觀察的事實 + 明確動作」，不得含「適當」「合理」「必要時」。
- 每份 rubric 末端必須有 fallback：「以上皆不適用時 → 升級」。弱模型遇到 rubric 沒涵蓋的情況，唯一正確動作是升級，不是自由發揮。

## Rubric 1：何時升級（隨時適用）

任一成立 → 停止修改、寫交接、升級：
- [ ] 同一測試／驗收連續失敗 2 次。
- [ ] 需要在多個方案間取捨，且本檔與 dispatch.md 沒給判準。
- [ ] 即將修改硬規則保護的檔案（見 CLAUDE.md 硬規則段）。
- [ ] 需求有兩種以上讀法，選錯代價高於來回確認的成本。
- [ ] 發現守則之間互相矛盾。
- [ ] 發現自己在「猜」而非「推導」——說不出從證據到結論的推理鏈。

## Rubric 2：何時算完成（宣告完成前逐項打勾）

- [ ] 三要素的驗收條件逐條執行過，結果原文附在回報中。
- [ ] `<專案測試指令>` 全綠（或明註哪些失敗是既有的，附證據如 git stash 前後對照）。
- [ ] 修 bug 類任務：附 fail-then-pass 證據——修復前該測試失敗、修復後通過的兩段輸出。只有綠燈而無紅燈在前，不能證明測試真的涵蓋了這個 bug。
- [ ] diff 範圍不超出任務目標；順手改的無關內容已還原。
- [ ] 新增的每個檔案落點符合專案慣例（對照鄰近同類檔案）。
- [ ] 回報使用指定格式，未加入「應該可以」「大致上」等模糊語。

## Rubric 3：何時停下（避免越陷越深）

任一成立 → 停下，回報現況而非繼續嘗試：
- [ ] 為了修一個問題而引入的改動已波及第 3 個模組。
- [ ] 修 A 壞 B、修 B 又壞 A 的蹺蹺板已來回兩次——停止局部修補，回頭重新定位根因。
- [ ] 開始想要註解掉測試、放寬斷言、或加 try/catch 吞錯誤來讓驗收通過。
- [ ] 同一個檔案在本 session 內已被自己改壞並還原過 2 次。
- [ ] 剩餘 context 明顯不足以完成加驗證，應改寫交接。

## Rubric 4：品質驗證（驗證者用）

依序檢查，第一個不通過即可退回：
1. 驗收條件是否逐條有執行證據？（無證據＝不通過，不論解釋多合理）
2. diff 是否含任務外的修改？
3. 是否違反 CLAUDE.md 硬規則或同步耦合規則？
4. 新程式碼與鄰近程式碼的風格（命名、註解密度、錯誤處理方式）是否一致？
5. 邊界情況：空輸入、重複執行、失敗路徑，是否至少各有一個測試或說明？

## Fallback

以上任何 rubric 無法涵蓋當前情況 → 視同 Rubric 1 成立，升級。
